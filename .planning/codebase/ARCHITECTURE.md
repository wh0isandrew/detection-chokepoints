# Architecture — detection-chokepoints

Data-driven static site. The source of truth is YAML files in `chokepoints/`; everything else — the rendered HTML, the search index, the collection stubs — is generated from them. The build pipeline is: Python pre-build → Jekyll build → GitHub Pages deploy.

## Build Pipeline

```
chokepoints/**/*.yml          sigma-rules/**/*.yml
         │                              │
         └──────────┐      ┌────────────┘
                    ▼      ▼
            scripts/aggregate.py
                    │
          ┌─────────┼──────────────┐
          ▼         ▼              ▼
  _data/            _chokepoints/  assets/js/
  chokepoints.yml   <slug>.md      search-index.json
  (Jekyll data)     (stubs)        (Fuse.js)
          │
          ▼
     jekyll build
          │
          ▼
        _site/   ──── GitHub Pages ──── public site
```

## Data Enrichment in aggregate.py

`aggregate.py` does more than copy YAML — it enriches each entry with computed fields before writing:

- `_tactic` — derived from directory name
- `_slug` — derived from filename stem
- `_sigma_<level>` — raw text of matching Sigma rule files, embedded inline
- `EarlyDetections[*]._rule_content` — raw text of referenced IOK/Sigma rules
- `_emulation_content` — raw text of the emulation script
- `_prerequisites_text`, `_variation_names` — flattened text for search indexing

This means templates can render rule code and emulation scripts without any file I/O — they read from `_data/chokepoints.yml`.

## Jekyll Collection Architecture

- **`_chokepoints/`** — thin stub `.md` files generated by `aggregate.py`; each has front matter `layout: chokepoint`, `slug`, `title`
- **`_data/chokepoints.yml`** — the full enriched data array; templates access it via `site.data.chokepoints`
- **`_layouts/chokepoint.html`** — master template for chokepoint detail pages; iterates over the entry's `Chokepoints`, `Variations`, `Detections`, `EarlyDetections`, etc.
- **`_layouts/attack-chain.html`** — template for attack chain pages
- **`_includes/`** — partials: `chokepoint-card.html`, `nav.html`, `ttp-overlap-diagram.html`, `ttp-vertical-diagram.html`, `attack-flow.html`

## Search Architecture

Two-tier search:
1. `scripts/aggregate.py` writes `assets/js/search-index.json` — a lean array of `{id, name, slug, tactic, mitreIds, detectionPriority, ...}`
2. `assets/js/search.js` loads the index and runs Fuse.js client-side fuzzy search

## YAML → HTML Contract

Each chokepoint YAML field maps to a specific rendering in `_layouts/chokepoint.html`:
- `Chokepoints[]` → numbered detection chain cards with stage/invariant/log sources
- `Variations[]` → variation table with status badges
- `Detections[]` + `_sigma_<level>` → tabbed maturity level cards with embedded rule code
- `EarlyDetections[]._rule_content` → pre-execution detection section
- `EvolutionTimeline[]` → timeline cards
- `OsintSources[]` → OSINT query cards
- `RawLogs[]` → sample log expanders
- `EmulationScript` + `_emulation_content` → emulation section

## Permalink Structure

- Chokepoint pages: `/detection-chokepoints/chokepoints/<slug>/`
- Index: `/detection-chokepoints/`

## CI/CD

`.github/workflows/pages.yml` — runs `python scripts/aggregate.py` then `jekyll build` on push to main; deploys `_site/` to GitHub Pages.
