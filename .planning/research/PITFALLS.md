# Pitfalls Research

Domain: Detection engineering knowledge base — OSINT pipeline, static site, writing/UX cleanup.

---

## Pitfall: Broken Pipeline Silently Publishes Stale or Empty Data

The masq-infra page displays whatever is in `_data/masq_infra.json` at build time. If the collection pipeline fails partway through, `aggregate.py` still runs cleanly and Jekyll still builds. The site goes live with stale or empty data with no visible indication anything is wrong.

**Warning signs:**
- `record_count` in `meta` doesn't increase across successive runs
- `last_updated` timestamp in `masq_infra.json` stops advancing
- `high_confidence_count` drops to zero without explanation
- CI passes but API response logs show 4xx/5xx errors

**Prevention:**
- Add a minimum-record guard in `build_data.py` — if record count falls below a configurable floor, abort the write and exit non-zero so CI fails visibly
- Write a `pipeline_status` key into `masq_infra.json` that the Liquid template can surface as a warning banner when the last run is older than N days
- Run the Streamlit validation harness over several days before wiring pipeline output into the site — this validation step must not be skipped

**Phase:** OSINT pipeline fix and validation (before any trends page goes live)

## Pitfall: Confidence Score Inflation from Structural Completeness

The scoring model awards points for having a SHA-256, a family name, and a domain. These fields are almost always present for URLhaus/MalwareBazaar records, so a large fraction of records will score 70+ and earn `high` or `confirmed` labels — not because the evidence is strong, but because the record is structurally complete.

**Warning signs:**
- More than 60% of published records carry `high` or `confirmed` labels
- Campaign clustering produces dozens of campaigns per run, most anchored only by `shared_ip`
- The Streamlit harness shows record labels flipping between runs for the same domain

**Prevention:**
- Separate structural completeness (`has_sha256`, `has_family`, `has_domain`) into a `data_quality` gate, distinct from `confidence`
- Reserve `confidence` for adversary-behavior evidence: confirmed payload delivery chain, multi-source corroboration, or analyst/Claude triage
- Require at least two independent source corroborations before publishing a `confirmed` label

**Phase:** OSINT pipeline fix and validation

## Pitfall: Partial API Runs Poison Historical Trend Data

Seven-source pipeline runs with no partial-run handling mean a mid-run rate limit or auth failure silently produces an incomplete `masq_infra.json` that gets committed to history and permanently skews trend lines. The `VALIDIN_CALL_BUDGET` is hardcoded at 100 — a known ceiling that can be reached before the infra hunt completes.

**Warning signs:**
- History shows a run with record count far below adjacent runs
- `sources` array in `meta` is missing feeds that should appear every run
- Shodan credit consumption logs show budget exceeded before all brands are queried

**Prevention:**
- Run sources in priority order; treat each source as independently optional
- Write partial record sets with a `partial_run: true` flag; do not append partial runs to history
- Add per-source timeout and retry logic with explicit failure logging before continuing to the next source
- Separate credit-constrained sources (URLScan, Shodan) from free feeds (URLhaus, MalwareBazaar)

**Phase:** OSINT pipeline fix and validation

## Pitfall: Cross-Source Field Mismatches Produce False Correlations

Each source has different field semantics the current shared normalizer doesn't account for. URLhaus `url_status` is not the same as ThreatFox `ioc_type`. MalwareBazaar `first_seen` is submission time, not first-observed-in-the-wild. `tldextract` on a URLhaus CDN-hosted URL extracts the CDN domain, not the attacker's domain.

**Warning signs:**
- Campaign clusters contain obvious CDN infrastructure (cloudflare-ipfs.com, githubusercontent.com)
- `lure_brand` extracted from URLhaus paths matches the CDN provider, not the impersonated software
- `domain` field in some records is an IP address string rather than a domain

**Prevention:**
- Write per-source normalization functions with documented field-mapping decisions, rather than a generic shared normalizer
- Validate extracted `domain` against a CDN/hosting provider blocklist before writing the record
- Prefer `first_seen` from URLScan scan history or ThreatFox over MalwareBazaar submission date
- Write normalization tests using captured API response fixtures before running against live APIs

**Phase:** OSINT pipeline fix and validation

## Pitfall: Generated Files in Git Create Misleading State

`_data/chokepoints.yml`, `_chokepoints/`, and `assets/js/search-index.json` are generated by `aggregate.py`. If they are committed and a YAML source edit is made without re-running aggregate.py, the site renders stale content. CI overwrites them on push, but local dev shows misleading state throughout the milestone.

**Warning signs:**
- `git diff` shows changes to `_data/chokepoints.yml` that don't correspond to any YAML source edit
- A chokepoint page shows old content in local `jekyll serve` despite a saved YAML edit
- Search returns results for content that has been removed

**Prevention:**
- Verify `_data/chokepoints.yml`, `_chokepoints/`, and `assets/js/search-index.json` are in `.gitignore` before starting any work — LEARNED.md says they should be but CONCERNS.md suggests they may not be
- If they are committed, remove them from tracking (`git rm --cached`) as the first commit of the milestone
- Add a pre-commit check that fails if any generated file is staged

**Phase:** First commit of the milestone, before any other work

## Pitfall: Liquid Template Errors Fail Silently

Jekyll's Liquid templating fails silently for many error conditions: a missing variable renders as empty string, a broken `forloop` skips all iterations. CHALLENGES.md documents a real instance: the `forloop.parentloop.index == 2` hardcoded stage matching produces wrong output for chokepoints with different stage counts.

**Warning signs:**
- A section of a chokepoint page is blank with no build error
- Detection tier badges appear on wrong stages
- A Sigma rule block is missing from a page that has the corresponding `.yml` file

**Prevention:**
- Replace positional index matching with explicit field-based matching (match on `Stage` name or `DetectionTier` value, not loop index)
- Add a post-build smoke test checking rendered HTML for expected section presence
- Run `jekyll build --strict` locally to surface Liquid warnings before pushing

**Phase:** Visual design and UX improvements (template changes carry this risk)

## Pitfall: Writing Pass Introduces Style Drift Across Pages

A site-wide writing pass across 8+ pages is high-surface-area work. Without a calibration reference, voice compounds: the writer applies one pattern on page 1, adjusts on page 3, and by page 8 the voice has drifted. CLAUDE.md defines the writing style but the primary reference is YAML, not prose.

**Warning signs:**
- Introductory paragraphs on later-edited pages are noticeably shorter or more formal than earlier-edited ones
- "The attacker must..." framing appears on some pages but not others
- Passive constructions appear only in later-edited pages

**Prevention:**
- Create a brief prose calibration reference (2-3 example paragraphs) at the start of the writing pass before editing begins
- Edit in a single session or review pass rather than across days
- Do a final cross-page pass comparing opening paragraphs across all pages before committing

**Phase:** Writing and content consistency pass

## Pitfall: CSS Scope Creep from Page-Specific Style Blocks

`trends/masq-infra.md` already has an inline `<style>` block. When the visual design pass adds new components, the path of least resistance is to add another inline block. Over several pages this produces duplicate rules, conflicting specificity, and components that look different across pages.

**Warning signs:**
- The same class (e.g., `.badge`) is defined in both `style.css` and a page-specific `<style>` block
- A navigation element looks slightly different on trends pages versus chokepoint pages
- More than 2-3 files have embedded `<style>` tags

**Prevention:**
- Enforce the badge reuse rule: always use `.badge` base class from `style.css`, never create standalone badge styles in page files
- Before adding any new CSS, check whether it belongs in `style.css` as a reusable class
- Audit existing inline `<style>` blocks at the start of the design phase and migrate reusable rules to `style.css`

**Phase:** Visual design and navigation/UX improvements

## Pitfall: Navigation or Heading Changes Break Deep Links

Chokepoint pages are referenced from Sigma rule `references` fields and the search index. A URL structure change, slug rename, or heading text change silently breaks inbound links. With a static site on GitHub Pages there is no redirect infrastructure.

**Warning signs:**
- Sigma rule `references` fields point to URLs that no longer resolve after a slug change
- The search index returns results that link to 404 pages

**Prevention:**
- Do not rename slugs or change URL structure during the UX pass; visual changes only
- If a heading must be renamed, check all internal `<a href="#...">` references in templates first
- Add explicit `id` attributes to critical section anchors so heading text can change without breaking links
- Run a link-check pass as the final step before merging any navigation-restructuring commit

**Phase:** Visual design and navigation/UX improvements

## Pitfall: Pipeline Schema Changes Silently Blank Trends Page Sections

The Liquid templates in `trends/masq-infra.md` are written against `_data/masq_infra_schema.md`. A pipeline fix that renames a field silently blanks every visualization using that field. The build passes and the page renders — just with empty sections.

**Warning signs:**
- A trends page section that rendered correctly is now blank after a pipeline update
- `aggregate.py` and `build_data.py` both succeed but the Liquid template iterates over an empty array

**Prevention:**
- Treat `masq_infra_schema.md` as binding: no field renames or removals without a corresponding template update in the same commit
- Add a schema validation step in `build_data.py` that checks assembled JSON against the schema before writing to `_data/`
- Have the Liquid template check `meta.schema_version` and surface a warning banner on mismatch

**Phase:** OSINT pipeline fix; respect again during visual design when templates are touched

## Key Findings

**Pipeline reliability is the highest-risk area.** The collection pipeline touches seven external APIs, uses a custom confidence scoring model with a structural-completeness conflation problem, and feeds a static site with no runtime fallback. Partial runs and silent failures are the most likely failure modes. The Streamlit validation requirement in PROJECT.md is the correct mitigation — it must not be skipped.

**Confidence scoring conflates data quality with threat certainty.** Records marked `confirmed` because they have a SHA-256 and a source name make the trends page misleading to a security-community audience. Fixing the scoring model before wiring data into the site is higher priority than visual polish.

**CSS and template risks are manageable with existing discipline.** The badge reuse rule and single `style.css` structure are good foundations. The risk is incremental scope creep during the design pass.

**Writing consistency is a process risk, not a technical one.** Voice drift compounds across pages when the pass runs over multiple sessions. A prose calibration reference created before editing begins prevents this.

**Resolve generated-files-in-git ambiguity before anything else.** CONCERNS.md flags this. Every other piece of work touches files that aggregate.py regenerates. Debugging on top of a confused git state will add noise throughout.
